# Lawbor项目实现方案

## 一、用户定位与市场洞察

目标用户锁定为**法学毕业生、求职新人**。团队基于法律实务经验，发现70%职场新人不懂劳动法、60%合同存在风险条款，而传统维权渠道成本高、周期长。市面上单一的法律智能体或工作流无法满足多元化需求，团队决定打造具有**公益价值**的Web应用，让广大学生群体通过浏览器即可获得法律守护。

## 二、产品形式与技术路线

选择Web应用形式，原因有三：①学生群体电脑使用频率高；②无需下载安装，降低使用门槛；③便于持续更新维护。定位"简约、现代、年轻化"，摒弃传统法律产品的沉稳刻板风格。

## 三、AI辅助开发流程

### 3.1 需求整理与分工
团队协作敲定定位、风格与功能需求，形成需求清单。开发过程中，**技术负责人全权负责提示词编写、环境变量配置、云平台部署**，确保技术栈统一与代码质量。

### 3.2 前端开发：外国AI生成简约UI
前端使用**Google AI Studio生成**，主要考量：外国AI设计的UI风格简约、留白合理、交互现代，契合年轻人审美。曾尝试国内大模型生成UI，但效果偏"体制内画风"，沉稳固定的设计可能劝退以年轻人为主力的求职群体。

### 3.3 后端开发：多AI协作
后端路由响应由**Claude Code、DuMate、WorkBuddy**共同完成，三者分工明确：Claude Code处理复杂逻辑、DuMate优化代码质量、WorkBuddy负责环境适配。

### 3.4 核心AI能力集成
- **DeepSeek大模型**：承担法律对话、合同审查等核心推理任务
- **Dify知识库API**：定向化识别劳动法垂类数据，提升专业性
- **企查查MCP服务**：企业风险查询（劳动纠纷、司法风险）
- **Google Web Speech API**：模拟面试语音识别，以最低成本实现实时交互

技术栈形成"**通用大模型+垂类知识库+专项服务**"的三层架构。

## 四、核心功能实现

### 4.1 智能合同审查
流程：用户上传合同 → PDF.js提取文字 → 正则提取公司名 → 企查查MCP查询风险 → DeepSeek逐条分析 → 生成风险等级（高/中/低）及法律建议。关键创新：**正则表达式宽松设计**，允许"甲方（用人单位）：XXX公司"等多样格式，配合多级fallback确保提取准确。

### 4.2 AI模拟面试
创新点：**温度值映射性格谱系**。用户可调节面试官性格（0-100），0为温和引导型、100为高压挑战型。DeepSeek根据温度值生成对应风格的追问逻辑，结合Web Speech API实现语音对话，让面试体验更真实。

### 4.3 税务计算器
难点：AI生成的税法计算公式存在滞后性。解决方案：**人工投喂最新公式**，前端内置计算逻辑，确保100%准确性。同时支持多收入来源（劳务报酬、稿酬等）的年度汇算清缴。

### 4.4 用户数据安全
采用**Supabase + Row Level Security**架构：用户认证由Supabase托管，数据表启用RLS策略，确保用户只能访问自己的对话历史、审查记录等敏感数据，实现端到端隐私保护。

## 五、挑战与应对

### 5.1 技术难点
| 问题 | 根因 | 解决方案 |
|------|------|----------|
| 路由响应链接不同 | React Router配置不当 | 统一使用BrowserRouter，Vercel配置rewrite |
| 前端静止无响应 | 组件状态未更新 | 引入useState管理loading状态 |
| 语音识别失败 | Chrome依赖Google服务被墙 | 添加网络检测+用户提示 |
| MCP数据解析失败 | 嵌套content数组格式 | JSON.parse提取+多格式兼容 |
| 本地vs生产差异 | Express与Serverless格式不同 | 维护server.ts（本地）和api/*.mjs（生产）双套代码 |

### 5.2 数据准确性保障
AI生成的法律内容需人工审核。团队建立"**AI生成初稿→法律专业成员校验→更新知识库**"闭环，确保法条引用准确、政策信息最新。

## 六、部署与效果

最终部署至**Vercel云端**，实现自动化CI/CD。技术负责人负责环境变量配置（Supabase、DeepSeek、企查查API Key等），确保生产环境安全运行。

**交付成果**：一站式劳动法保护网站，包含合同审查、AI对话、模拟面试、税务计算、补贴匹配、仲裁指南6大功能。界面简约现代，响应速度快（首屏<1.5秒），支持中英双语，数据持久化。

**核心价值**：以"劳友记"理念，用手托举职场新人，让懂法、用法、维权不再困难。
